## TODO 3bis — Garde-fous quota auto-stop (À FAIRE EN MÊME TEMPS QUE TODO 3)

**Philosophie** : Arsenal ne doit pas bouffer 100 % du quota Pro Max au point
de m'empêcher d'utiliser Claude Code pour autre chose.

Deux seuils indépendants, configurables dans la GUI, avec valeurs par défaut
sensées (à confirmer avec user) :

### Seuil SESSION (défaut suggéré : 70 %)

- Affichage : barre de progression "Session : XX % (Y h Zm avant reset)"
  avec couleur dynamique :
  - vert si < seuil - 20
  - orange si entre seuil - 20 et seuil
  - rouge si >= seuil
- Quand `session_used / session_limit >= seuil` :
  - Auto-Stop d'Arsenal (même chemin que clic Stop : Ctrl-Break → embed
    Discord "Interrompu — quota session atteint" → release_lock).
  - Flag `_throttle_session` activé en mémoire.
  - **Auto-resume** : la GUI continue de poller le quota toutes les 60 s ;
    quand le reset session est passé (timestamp `reset_at` dépassé OU
    `session_used` redescendu), flag levé + relance auto si auto-restart
    est coché.
- L'utilisateur peut désactiver le throttle session manuellement (bouton
  "Forcer continue malgré seuil" qui met le flag à False jusqu'au prochain
  Stop manuel).

### Seuil HEBDOMADAIRE (défaut suggéré : 80 %)

- Affichage : barre de progression "Hebdo : XX % (J jours avant reset)".
- Quand `weekly_used / weekly_limit >= seuil` :
  - Auto-Stop d'Arsenal (même chemin que session).
  - Flag `_throttle_weekly` activé **et PERSISTANT** : sauvegardé dans
    `_secrets/quota_state.json` pour survivre à un restart de la GUI.
  - Auto-restart **désactivé** tant que ce flag est levé.
  - Bouton ▶ Lancer **disabled** avec tooltip "Limite hebdo X % atteinte,
    réajuster le seuil ou attendre le reset (J jours)".
  - Embed Discord rouge dans #logs : "🚨 Limite hebdo Arsenal atteinte
    (XX % de YY %), pause forcée jusqu'au DD/MM ou réajustement manuel".
- Pour reprendre, l'utilisateur doit explicitement :
  - augmenter le seuil dans la GUI (slider qui déclenche un re-check), OU
  - cliquer "Forcer reset throttle hebdo" (bouton rouge, demande
    confirmation), OU
  - attendre le reset auto (la GUI re-poll et lève le flag quand
    `weekly_used / weekly_limit < seuil`).

### Implémentation concrète à faire

1. Étendre `claude_usage.py` avec un dataclass `Quota` :
   ```python
   @dataclass
   class Quota:
       session_used: int
       session_limit: int
       weekly_used: int
       weekly_limit: int
       session_reset_at: datetime
       weekly_reset_at: datetime

       @property
       def session_pct(self) -> float: ...
       @property
       def weekly_pct(self) -> float: ...
   ```
2. Dans `summarize_gui.py`, ajouter une nouvelle frame "Quota Pro Max"
   AVANT la frame "Console", contenant :
   - 2 ttk.Progressbar (session + hebdo) avec label % + temps reset
   - 2 Spinbox/Scale pour les seuils (session 0-100, hebdo 0-100), avec
     persistance dans `_secrets/quota_state.json`
   - Indicateur d'état du throttle (icône verte / orange / rouge)
   - Bouton "Forcer reset throttle hebdo" (visible seulement si flag levé)
3. Boucle de poll : `root.after(60_000, _refresh_quota_and_check)` qui :
   - appelle `fetch_usage()`
   - update les barres
   - check les seuils et trigger auto-stop si dépassement
   - update l'état du throttle persistant
4. Persistance `_secrets/quota_state.json` (atomic write `.tmp + os.replace`) :
   ```json
   {
     "session_threshold_pct": 70,
     "weekly_threshold_pct": 80,
     "weekly_throttled": false,
     "weekly_throttled_at": null,
     "last_check_at": "2026-04-29T08:00:00Z"
   }
   ```
